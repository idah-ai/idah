# Media Service

## Overview

The Media Service handles **file storage, retrieval, and media processing** across IDAH. It manages raw file uploads, serves files via streaming, and orchestrates modality-specific processing (e.g., video transcoding, image thumbnail generation) when entries are created on datasets.

All file storage uses **Shrine** with an injectable storage backend (local filesystem in development, S3/GCS in production). Processing jobs are scheduled via the same `Jobs::Scheduler` infrastructure shared with the Sync Service.

---

## Models

### Medias::Record

Shrine-backed file record stored in the `medias` table.

| Field | Type | Description |
|-------|------|-------------|
| `id` | `uuid` | Primary key (UUIDv7) |
| `resource` | `text` | Logical resource type grouping (e.g. `uploads`, `thumbnails`, `exports`) |
| `key` | `text` | Shrine storage key (auto-generated by Shrine; `nil` for top-level resource queries) |
| `filename` | `text` | Original uploaded filename |
| `size` | `bigint` | File size in bytes |
| `mime_type` | `text` | Detected MIME type |
| `created_by` | `uuid` | Account ID that created the record |
| `created_role` | `text` | Role name of the creating account |
| `public` | `boolean` | Whether the file is publicly accessible without auth |
| `meta` | `jsonb` | Arbitrary metadata key-value store |
| `project_id` | `uuid` | Optional project association |
| `created_at` | `timestamptz` | Creation timestamp |
| `updated_at` | `timestamptz` | Last update timestamp |

The combination of `resource` + `key` forms the primary lookup path. When `key` is `nil`, the record represents a resource directory rather than a single file.

#### Shrine Integration

```ruby
class Medias::Record < Sequel::Model(:medias)
  include Verse::Model

  # Shrine attachment — allows `file_data` column management
  include Medias::Uploader::Attachment.new(:file)

  plugin :timestamps, update_on_create: true
end
```

The `Medias::Uploader` is initialized with the `determine_mime_type` plugin:

```ruby
class Medias::Uploader < Shrine
  plugin :determine_mime_type, analyzer: :marcel
end
```

Storage is injected at boot via `Verse::Plugin[:shrine].with_storage(:store)` and can be toggled between local disk and cloud backends.

### Jobs::Record

Standard job record — identical to the schema described in [`guide/job-system.md`](guide/job-system.md).

| Field | Type | Description |
|-------|------|-------------|
| `id` | `uuid` | Primary key (UUIDv7) |
| `type` | `text` | Job type discriminator (e.g. `processing`, `export`) |
| `status` | `text` | Current status: `pending`, `running`, `completed`, `failed` |
| `metadata` | `jsonb` | Arbitrary job payload/metadata |
| `scheduled_at` | `timestamptz` | When the job should be picked up |
| `started_at` | `timestamptz` | When processing started |
| `completed_at` | `timestamptz` | When processing finished |
| `error` | `jsonb` | Error details on failure |
| `created_at` | `timestamptz` | Creation timestamp |
| `updated_at` | `timestamptz` | Last update timestamp |

## Endpoints

### HTTP — `/medias`

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `GET` | `/medias/files/:resource` | Authenticated | List files for a resource group |
| `GET` | `/medias/files/:resource/:key` | Authenticated | Fetch file metadata |
| `POST` | `/medias/files/:resource(/:key)?` | Authenticated | Upload a file (multipart) |
| `GET` | `/medias/files/:resource/:key/download` | Authenticated | Stream file content |

#### Upload Flow

```
POST /medias/files/:resource(/:key)?
  → Medias::Service.upload(resource:, key: nil, file:)
    1. Shrine upload (file_data column populated)
    2. DB record created in transaction
    3. On failure: Shrine rollback (delete uploaded file)
  → Returns JSON:API Medias::Record
```

The `upload` method executes inside a `DB.transaction` with a rollback safeguard:

```ruby
def upload(resource:, key: nil, file:)
  uploaded_file = uploader.upload(file)
  record = repository.create!(
    resource: resource,
    key: uploaded_file.id,
    filename: uploaded_file.original_filename,
    size: uploaded_file.size,
    mime_type: uploaded_file.mime_type
  )
  record
rescue => e
  uploaded_file&.delete # rollback Shrine storage
  raise
end
```

#### Download Flow

```
GET /medias/files/:resource/:key/download
  → Medias::Service.download(resource:, key:)
    1. Look up Medias::Record by resource + key
    2. Open Shrine storage file handle
    3. Return streaming body
  → Rack streaming response
```

The download endpoint returns a `Rack::Body::Proxy` that streams the file chunk-by-chunk without loading it fully into memory:

```ruby
def download(resource:, key:)
  record = repository.find_by_resource_and_key!(resource, key)
  file = record.file # Shrine::UploadedFile
  streaming_body(file)
end

private

def streaming_body(file)
  env = {
    "Content-Type" => file.mime_type,
    "Content-Disposition" => %(attachment; filename="#{file.original_filename}"),
    "Content-Length" => file.size.to_s
  }
  [env, [file.read]] # Simplified — actual impl uses streaming enumerator
end
```

| Event | Direction | Trigger | Description |
|-------|-----------|---------|-------------|
| `dataset:entries:created` | **Subscribed** | Dataset Service | Entry created on a dataset → trigger media processor |
| `media:medias:created` | **Published** | Media Service | File uploaded → downstream services notified |
| `media:medias:updated` | **Published** | Media Service | File metadata changed |
| `media:medias:deleted` | **Published** | Media Service | File removed |

```ruby
class Medias::Service < Verse::Service::Base

  end

```

Key design points:

- **Upload transactions**: Upload to Shrine first, then create DB record inside a transaction. If the DB write fails, roll back the Shrine upload.
- **Auth scoping**: Repository scopes queries by the authenticated user's project memberships.
- **MIME detection**: Shrine's `determine_mime_type` plugin runs on upload to ensure `mime_type` is accurate.

### Jobs::Service

Standard job CRUD — see [`guide/job-system.md`](guide/job-system.md) for full documentation.

### Jobs::Scheduler

Background scheduler that polls for pending/running jobs. Uses `MonitorMixin` for thread safety. See [`guide/job-system.md`](guide/job-system.md).

### Jobs::Base

Base class for all job implementations. Provides `run` interface and status tracking.

### Processor::Service

Orchestrates entry processing when entries are created or updated.

```ruby
class Processor::Service < Verse::Service::Base
  # Triggered by dataset:entries:created event
  def process_entry(entry_id:, dataset_id:, modality_id:)
    processor_class = Processor::Registry.for(modality_id)
    return unless processor_class # No processor registered for this modality

    # Wrap processor in a Job, schedule it
    Jobs::Service.create(
      type: "processing",
      metadata: {
        processor: processor_class.name,
        entry_id: entry_id,
        dataset_id: dataset_id,
        modality_id: modality_id
      }
    )
  end
end
```

The flow:

1. Dataset Service publishes `dataset:entries:created` event
2. Media Service event handler receives the event
3. `Processor::Service.process_entry` looks up the modality in the registry
4. If a processor exists, it wraps the processor call in a `Jobs::Record` and schedules it
5. The `Jobs::Scheduler` picks up the job and runs the `Processor::Job`

### Processor::Registry

Maps modality IDs to processor classes. Populated at boot by plugins and built-in processors.

```ruby
class Processor::Registry
  class << self
    def register(modality_id, processor_class)
      processors[modality_id] = processor_class
    end

    def for(modality_id)
      processors[modality_id]
    end

    def processors
      @processors ||= {}
    end
  end
end
```

Registering a new processor (either from a plugin or from core code):

```ruby
Processor::Registry.register("idah-video", Video::Service)
```

### Processor::Job

Wraps a processor invocation inside the `Jobs::Base` framework.

```ruby
class Processor::Job < Jobs::Base
  def run
    context = Processor::Context.new(
      job:,
      entry_id: metadata[:entry_id],
      dataset_id: metadata[:dataset_id]
    )
    processor_class = Processor::Registry.for(metadata[:modality_id])
    processor_class.new(context).call
  end
end
```

### Processor::Context

Provides processors with controlled access to the media system:

- **`context.download(file_key)`** — Download a source file for processing
- **`context.upload(filename:, mime_type:, data:)`** — Upload a processed result
- **`context.media_metadata(key)`** — Look up media record metadata
- **`context.job`** — The current job record (for progress reporting)

The context ensures processors cannot bypass auth or storage boundaries.

### Video::Service

Built-in processor for video modality entries. Registered as the default processor for modalities matching `/video/`.

```ruby
class Video::Service
  def initialize(context)
    @context = context
  end

  def call

    # 2. Run ffmpeg transcoding/spriting via Executor
    # 3. Upload result thumbnails/sprite sheets via context.upload

end
```

Shell command runner backed by a thread pool.

## DB Migrations

The Media Service has a single migration file that creates both the `medias` and `jobs` tables, plus UUID v7/v8 generation functions (shared across services where needed).

### Migration: `001_create_media_and_jobs.rb`

```ruby
Sequel.migration do
  up do
    # UUID v7 generation function (RFC 9562)
    run <<~SQL
      CREATE OR REPLACE FUNCTION uuid_generate_v7() RETURNS uuid AS $$
        SELECT encode(
          set_byte(
            set_byte(
              overlay(
                gen_random_bytes(16) placing
                (floor(extract(epoch from clock_timestamp()) * 1000)::bigint::bytea)
                FROM 1 FOR 6
              ),
              7, (b'0111'::bit(4))::integer
            ),
            9, (b'10'::bit(2))::integer
          ),
          'hex'
        )::uuid;
      $$ LANGUAGE SQL VOLATILE;
    SQL

    create_table(:medias) do
      column :id, :uuid, default: Sequel.lit("uuid_generate_v7()"), primary_key: true
      String :resource, null: false
      String :key
      String :filename, null: false
      Bignum :size, null: false
      String :mime_type, null: false
      column :created_by, :uuid
      String :created_role
      TrueClass :public, default: false
      column :meta, :jsonb, default: Sequel.lit("'{}'::jsonb")
      column :project_id, :uuid

      DateTime :created_at, null: false, default: Sequel::CURRENT_TIMESTAMP
      DateTime :updated_at, null: false, default: Sequel::CURRENT_TIMESTAMP

      index [:resource, :key], unique: true, where: Sequel.lit("key IS NOT NULL")
      index :resource
      index :project_id
    end

    create_table(:jobs) do
      column :id, :uuid, default: Sequel.lit("uuid_generate_v7()"), primary_key: true
      String :type, null: false
      String :status, null: false, default: "pending"
      column :metadata, :jsonb, default: Sequel.lit("'{}'::jsonb")
      DateTime :scheduled_at, null: false, default: Sequel::CURRENT_TIMESTAMP
      DateTime :started_at
      DateTime :completed_at
      column :error, :jsonb

      DateTime :created_at, null: false, default: Sequel::CURRENT_TIMESTAMP
      DateTime :updated_at, null: false, default: Sequel::CURRENT_TIMESTAMP

      index :status
      index [:status, :scheduled_at]
    end
  end

  down do
    drop_table(:medias)
    drop_table(:jobs)
    run "DROP FUNCTION IF EXISTS uuid_generate_v7()"
  end
end
```

Notes:

- The `uuid_generate_v7()` function is idempotent — subsequent migrations from other services can safely redefine it.
- The `key IS NOT NULL` partial unique index ensures that only records with explicit keys are unique per resource.
- The `jobs` table uses the same schema as the Sync Service's migration — if both services share a database, only one migration should create this table.

## Event Flow: Processing Entry

```
Dataset Service                    Media Service
───────────────                    ──────────────
Entry created
  → publishes
    dataset:entries:created
                              → Event handler receives event
                              → Processor::Service.process_entry
                                → Processor::Registry.for(modality_id)
                                → Processor::Job created + scheduled
                              → Jobs::Scheduler picks up job
                                → Processor::Job#run
                                  → Processor::Context setup
                                  → Video::Service#call (or other)
                                    → Executor.run("ffmpeg ...")
                                    → context.upload(result)
```
